# Agent Notes

This repository is a bkmpw-managed modpack source tree for Create Delight Project Rebirth.

Shared repository skills live under `.agents/skills/`.

- For pack metadata maintenance, read `.agents/skills/packwiz-modpack/SKILL.md` before changing pack metadata.
- For broader repository conventions, read `docs/DevGuide.md`.
- Launcher setup docs live in `docs/GettingStarted-HMCL.md`, `docs/GettingStarted-PCL.md`, and `docs/GettingStarted-Prism.md`.

## Hard Rules

- Do not commit mod jars, downloaded shader/resource pack zips, generated NeoForge libraries, worlds, logs, or runtime state.
- Do not vendor pack management binaries. Install `bkmpw` globally from npm with `npm install -g @bro-know-my/packwiz`.
- Do not reintroduce `scripts/bin/packwiz.exe`, `packwiz-old.exe`, `packwiz-installer-bootstrap.jar`, or their VERSION files.
- Track bkmpw/packwiz-style descriptors such as `mods/common/*.pw.toml`, `mods/client/*.pw.toml`, `mods/server/*.pw.toml`, `resourcepacks/*.pw.toml`, and `shaderpacks/*.pw.toml`. Root `pack.toml`, `index.toml`, and `.packwizignore` are generated by `devtool.bat prepare-pack` / `refresh`; keep the tracked templates in `pack/`.
- Prefer the root `devtool.bat` entry point on Windows, or `./devtool.sh` on Linux/macOS, for local bkmpw operations in this repository.
- Keep Minecraft `1.21.1`, NeoForge `21.1.228`, and Java `21` aligned across `pack/pack.toml`, `pack/variables.txt`, and documented examples.
- Treat `Create-Delight-Remake` as source reference only. Do not bulk-copy old Forge `1.20.1` KubeJS/configs into this repo without checking target mod availability and schema/API changes.

## Migration Approach

1. Maintain repository structure and pack metadata first.
2. Add or update mods through bkmpw descriptors.
3. Port configs from a working 1.21.1 NeoForge instance where possible.
4. Port KubeJS by layer: utilities, registrations, tags, recipes, client scripts, then optional integrations.
5. Verify with server boot or focused reload after each layer.
